# 퍼블릭 배포 가이드

## 메타
- 문서 목적: standalone package를 외부 접근 가능한 서버에 올리고, Vercel 또는 외부 대시보드가 HTTPS로 AI 프록시를 호출하도록 구성합니다.
- 대상 독자: 배포 담당자, 인프라 운영자, 프런트엔드 배포 담당자
- 기준 운영안: `kits/myagent-copilot-kit/standalone-package`
- 현재 기준 버전/산출물: `v0.1.0`, `release/myagent-copilot-standalone-v0.1.0.zip`
- 관련 문서:
  - `01-아키텍처-개요.md`
  - `05-보안-운영-가이드.md`
  - `07-검증-체크리스트.md`
  - `../standalone-package/OPERATIONS.md`

## 1. 목표 상태
퍼블릭 배포의 목표는 아래와 같습니다.
- 프록시는 외부 서버에서 상시 실행된다.
- 프록시는 HTTPS 뒤에서 동작한다.
- 프런트는 `https://api.your-domain.com/api/ai/chat`로만 호출한다.
- `MYAGENT_PROXY_AUTH_TOKEN`으로 무인증 외부 호출을 막는다.
- `MYAGENT_PROXY_CORS_ORIGINS`에 실제 대시보드 origin만 허용한다.

## 2. 권장 배포 구조
```text
사용자 브라우저
   |
   | HTTPS
   v
Vercel 대시보드
   |
   | HTTPS + x-ai-proxy-token
   v
api.your-domain.com
   |
   v
myagent-copilot-standalone (0.0.0.0:3010)
```

## 3. 서버 준비

### 3.1 필수 항목
- Node 22.12.0 이상
- pnpm 설치
- 외부에서 접근 가능한 서버
- HTTPS termination 수단
- 방화벽 규칙

### 3.2 권장 항목
- systemd, pm2, NSSM 같은 프로세스 관리기
- 리버스 프록시(Nginx, Caddy, Traefik)
- 로그 수집 경로

## 4. 환경변수
퍼블릭 배포에서는 아래 환경변수를 기본값으로 취급합니다.

```bash
MYAGENT_PROXY_HOST=0.0.0.0
MYAGENT_PROXY_PORT=3010
MYAGENT_PROXY_AUTH_TOKEN=change-me
MYAGENT_PROXY_CORS_ORIGINS=https://your-dashboard.vercel.app,https://dashboard.your-domain.com
MYAGENT_PROXY_ALLOW_SANITIZED_TO_COPILOT=0
```

설명:
- `MYAGENT_PROXY_AUTH_TOKEN`
  - 프런트가 호출할 때 같이 보내는 공유 토큰
- `MYAGENT_PROXY_CORS_ORIGINS`
  - 허용 origin의 콤마 구분 목록
- `MYAGENT_PROXY_ALLOW_SANITIZED_TO_COPILOT`
  - sanitize된 payload를 Copilot에 보내도 되는지 여부

## 5. 빌드 및 실행

### 5.1 소스 기준
```bash
cd kits/myagent-copilot-kit/standalone-package
pnpm install
pnpm build
node dist/cli.js serve --host 0.0.0.0 --port 3010
```

### 5.2 배치 파일 기준
Windows 서버라면:
```powershell
.\serve-public.bat
```

### 5.3 zip 배포 기준
1. `release/myagent-copilot-standalone-v0.1.0.zip`를 서버에 복사
2. 압축 해제
3. `pnpm install --frozen-lockfile`
4. `pnpm build`
5. 환경변수 설정
6. `node dist/cli.js serve --host 0.0.0.0 --port 3010`

## 6. 리버스 프록시와 HTTPS

### 6.1 원칙
- 브라우저에서 `http://127.0.0.1:3010`을 सीधे 호출하는 구조는 퍼블릭 배포에서 성립하지 않습니다.
- 반드시 공개 HTTPS 주소를 브라우저가 호출해야 합니다.

### 6.2 권장 예시
- 공개 주소: `https://api.your-domain.com`
- 내부 프록시: `http://127.0.0.1:3010`

## 7. 프런트엔드 설정

### 7.1 런타임 주입
```html
<script>
  window.__JPT71_AI_ENDPOINT__ = "https://api.your-domain.com/api/ai/chat";
  window.__JPT71_AI_PROXY_TOKEN__ = "replace-at-runtime";
</script>
```

### 7.2 fetch 예시
```ts
await fetch("https://api.your-domain.com/api/ai/chat", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "x-request-id": crypto.randomUUID(),
    "x-ai-proxy-token": window.__JPT71_AI_PROXY_TOKEN__,
    "x-ai-sensitivity": "internal",
  },
  body: JSON.stringify({
    model: "github-copilot/gpt-5-mini",
    sensitivity: "internal",
    messages: [
      { role: "system", content: "입력 JSON 밖 사실은 단정하지 말라." },
      { role: "user", content: "현재 KPI 기준 실행 액션을 3단계로 정리하라." },
    ],
  }),
});
```

## 8. 퍼블릭 배포에서 가장 많이 틀리는 부분
- origin 허용 목록에 실제 Vercel URL을 넣지 않음
- 토큰 인증 없이 `0.0.0.0`로 공개
- HTTPS 없이 브라우저에서 혼합 콘텐츠로 막힘
- sanitize payload를 Copilot에 보낼지 정책이 정해지지 않음
- HTML 리포트가 여전히 `127.0.0.1` 엔드포인트를 가리킴

## 9. 검증 순서
1. 서버에서 `GET /api/ai/health`
2. 허용 origin으로 `OPTIONS /api/ai/chat`
3. 정상 토큰으로 `POST /api/ai/chat`
4. 잘못된 origin으로 차단 확인
5. 잘못된 토큰으로 `401` 확인
6. 대시보드에서 실제 렌더 확인

## 운영 공통 블록
### 검증 명령
```bash
node dist/cli.js health
curl https://api.your-domain.com/api/ai/health
```

### 주의사항
- 퍼블릭 운영에서 프런트에 고정 공유 토큰을 넣는 구조는 최소 수준 보안입니다. 가능하면 서버 측 세션 또는 단기 서명 토큰으로 올리는 것이 맞습니다.
- `MYAGENT_PROXY_AUTH_TOKEN`, `MYAGENT_PROXY_CORS_ORIGINS` 없이 외부 공개 운영을 시작하지 마십시오.
