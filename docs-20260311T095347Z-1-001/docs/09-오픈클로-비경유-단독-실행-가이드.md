# 오픈클로 비경유 단독 실행 가이드

## 메타
- 문서 목적: `OpenClaw 제품 비경유`와 `standalone package 직접 실행`의 의미를 구분하고, 현재 기준 운영안이 무엇인지 명확히 설명합니다.
- 대상 독자: 기존 OpenClaw 사용자, standalone package 도입자, 이관 담당자
- 기준 운영안: `kits/myagent-copilot-kit/standalone-package`
- 현재 기준 버전/산출물: `v0.1.0`, `release/myagent-copilot-standalone-v0.1.0.zip`
- 관련 문서:
  - `../standalone-package/README.md`
  - `../standalone-package/MIGRATION.md`
  - `../../OpenClaw Copilot 단일 Provider.md`

## 1. 용어부터 정리

### 1.1 OpenClaw 제품 비경유
의미:
- `openclaw` CLI
- `openclaw gateway`
- OpenClaw 앱/세션/채널
를 실행하지 않고 AI 프록시를 직접 띄우는 방식입니다.

### 1.2 standalone package 직접 실행
의미:
- `kits/myagent-copilot-kit/standalone-package`
안의 자체 `package.json`, `src`, `dist`, `.bat`로만 실행하는 방식입니다.

현재 기준 운영안은 이 둘을 합친 것입니다. 즉, `OpenClaw 제품 비경유`이면서 `standalone package 직접 실행`입니다.

## 2. 무엇이 독립되었는가
- 실행 명령이 OpenClaw CLI에 의존하지 않음
- 상태 기본 경로가 `~/.myagent-copilot`로 분리됨
- `pnpm build`, `pnpm login`, `pnpm serve:*`가 자체 제공됨
- `dist/cli.js`와 `.bat` 파일로 직접 실행 가능

## 3. 무엇은 아직 호환으로 남아 있는가
- OpenClaw auth store fallback 읽기
  - `~/.openclaw/agents/main/agent/auth-profiles.json`
  - `~/.openclaw/auth-profiles.json`

즉, 현재 구조는 `완전 절연`이 아니라 `기본은 독립, 인증은 호환 fallback 가능` 구조입니다.

## 4. 직접 실행 명령

### 4.1 개발 실행
```bash
pnpm dev
```

### 4.2 빌드 후 실행
```bash
pnpm build
node dist/cli.js serve --host 127.0.0.1 --port 3010
```

### 4.3 로그인
```bash
pnpm login
```

### 4.4 토큰/사용량
```bash
pnpm token
pnpm usage
```

## 5. OpenClaw와 함께 둘 수 있는가
가능합니다. 다만 원칙은 다음과 같습니다.
- 운영 기준 문서는 standalone package를 따릅니다.
- OpenClaw는 호환/이관/비교 경로로만 남깁니다.
- 장기적으로는 standalone auth store를 기본으로 씁니다.

## 6. 왜 Standalone 우선인가
- 문서 기준이 단순해집니다.
- 대시보드 프로젝트에서 OpenClaw 전체를 띄우지 않아도 됩니다.
- 배포 단위가 작아집니다.
- 채널, 노드, 브라우저, gateway 기능과 분리되어 신뢰 경계가 명확해집니다.

## 7. 언제 OpenClaw 경유를 유지할 것인가
- 기존 auth/session 운영 체계가 이미 OpenClaw 중심일 때
- 채널 통합까지 같이 써야 할 때
- 아직 standalone으로 완전히 이관하기 전 과도기일 때

이 경우에도 기준 표현은 `OpenClaw 경유는 호환 경로`입니다.

## 운영 공통 블록
### 검증 명령
```bash
node ../standalone-package/dist/cli.js
node ../standalone-package/dist/cli.js health
```

### 주의사항
- OpenClaw auth store fallback이 동작한다고 해서 standalone auth store 구성을 미루지 마십시오. 운영 단순성과 감사 대응을 위해 별도 store로 수렴시키는 것이 좋습니다.
