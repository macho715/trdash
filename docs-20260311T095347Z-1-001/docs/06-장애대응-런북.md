## 1. 우선 확인 공통 절차
1. 프록시 프로세스가 살아 있는지 확인
2. `GET /api/ai/health` 확인
3. origin, auth token, request payload 확인
4. DLP/routing policy에 걸렸는지 확인
5. GitHub auth와 runtime token 상태 확인

## 2. 증상별 대응

### 2.1 `AI proxy unavailable`
원인 후보:
- 프로세스 미기동
- 포트 충돌
- 프런트가 잘못된 엔드포인트 호출

확인:
```bash
node dist/cli.js health
curl http://127.0.0.1:3010/api/ai/health
```

조치:
- 프록시 재기동
- 포트 `3010` 충돌 해소
- 프런트의 `AI_ENDPOINT` 확인

### 2.2 `CORS_ORIGIN_DENIED`
원인:
- `MYAGENT_PROXY_CORS_ORIGINS`에 현재 대시보드 주소가 없음

조치:
- 실제 origin을 정확히 추가
- 프로토콜, 포트, 서브도메인까지 일치 여부 확인

### 2.3 `PROXY_AUTH_REQUIRED`
원인:
- `MYAGENT_PROXY_AUTH_TOKEN`이 설정되어 있는데 프런트가 토큰을 안 보냄
- 잘못된 토큰 사용

조치:
- `x-ai-proxy-token`
- 또는 `Authorization: Bearer ...`
중 하나로 동일 값 전송

### 2.4 `DLP_BLOCKED`
원인:
- payload에 high/critical 민감 정보 포함

조치:
- 원문 데이터에서 비밀값 제거
- 요약 JSON만 보내도록 프롬프트 구성 변경
- 정말 필요한 경우 local-only route 구성

### 2.5 `DLP requires sanitization and policy forbids sending sanitized payloads to Copilot`
원인:
- medium 이상 민감 정보 포함
- `MYAGENT_PROXY_ALLOW_SANITIZED_TO_COPILOT=0`

조치:
- payload를 더 줄임
- 민감 값을 제거
- local route 도입
- 운영 정책상 허용 시에만 sanitize 전송 허용

### 2.6 `LOCAL_ROUTE_REQUIRED`
원인:
- 민감도가 `secret`
- local runner 미구성

조치:
- `sensitivity`를 실제 값에 맞게 낮추거나
- local runner를 구현하거나
- secret 데이터를 아예 외부 전송하지 않는 구조로 변경

### 2.7 GitHub 인증 실패
원인:
- standalone auth store 부재
- OpenClaw fallback auth store만 남아 있거나 만료
- 환경변수 토큰 만료

확인:
```bash
pnpm token
pnpm usage
```

조치:
- `pnpm login` 재실행
- `~/.myagent-copilot/auth-profiles.json` 확인

### 2.8 `429` 또는 rate limit
원인:
- Copilot 측 rate limiting

조치:
- 즉시 재시도 남발 금지
- 요청 간격 조정
- 실험/배치 작업 분리
- 운영 로그로 발생 시간대를 기록

## 3. 롤백 기준
아래 조건 중 하나면 퍼블릭 운영을 내리고 로컬 또는 제한된 운영으로 롤백하는 것이 맞습니다.
- sanitize 정책과 실제 데이터 구조가 계속 충돌
- 인증 토큰 유출 가능성이 의심됨
- origin allowlist를 명확히 관리할 수 없음
- OpenClaw fallback auth와 standalone auth가 충돌하여 운영 혼선이 큼

## 4. 긴급 복구 우선순위
1. 외부 공개 차단
2. 토큰 회전

4. 프록시 재기동
5. 인증 재설정
6. 프런트 런타임 설정 갱신
