## 1. 전체 구성
기본 구성은 아래와 같습니다.

```text
브라우저/대시보드/HTML 리포트
        |
        |  POST /api/ai/chat
        v
myagent-copilot-standalone proxy
  - CORS 검사
  - 프록시 토큰 인증
  - DLP 검사
  - 민감도 라우팅
  - Copilot prompt 실행
        |
        |  GitHub token -> runtime token exchange
        v
GitHub Copilot runtime endpoint
```

보조 경로는 다음과 같습니다.
- device login: GitHub device flow
- usage 조회: `GET https://api.github.com/copilot_internal/user`
- runtime token 교환: `GET https://api.github.com/copilot_internal/v2/token`

## 2. 구성 요소별 역할

### 2.1 브라우저/대시보드
- 역할: 사용자 질문, 프리셋 프롬프트, 필터 요약 JSON 전송
- 금지: 원본 대용량 데이터 전체 덤프, 장기 비밀값 저장, 프록시 토큰 하드코딩
- 권장: 필터/집계 요약만 payload에 포함

### 2.2 standalone proxy
- 역할: 단일 공인 API 표면 제공
- 공인 엔드포인트:
  - `GET /api/ai/health`
  - `POST /api/ai/chat`
- 기본 모델:
  - `github-copilot/gpt-5-mini`
- 부가 기능:
  - origin allowlist
  - optional `x-ai-proxy-token`
  - pre-send DLP
  - 민감도 기반 local/coplan/copilot route 결정
  - 운영 로그 출력

### 2.3 GitHub device login
- 역할: 사용자 GitHub 인증 토큰 확보
- 저장 위치:
  - `~/.myagent-copilot/auth-profiles.json`
- fallback:
  - `~/.openclaw/agents/main/agent/auth-profiles.json`
  - `~/.openclaw/auth-profiles.json`

### 2.4 runtime token cache
- 역할: Copilot API 호출용 짧은 수명 토큰 재사용
- 저장 위치:
  - `~/.myagent-copilot/cache/github-copilot.token.json`
- 재발급 기준:
  - 만료 5분 전부터 새 토큰 교환

## 3. 요청 처리 순서

### 3.1 `POST /api/ai/chat`
1. 브라우저가 `messages`, `model`, `sensitivity`를 전송합니다.
2. 프록시는 origin을 검사합니다.
3. 퍼블릭 모드라면 `x-ai-proxy-token` 또는 `Authorization: Bearer ...`를 검사합니다.
4. DLP가 메시지를 검사합니다.
5. DLP가 `block`이면 즉시 거부합니다.
6. DLP가 `sanitize`이고 정책상 Copilot 전송이 금지되어 있으면 local route 또는 차단으로 전환합니다.
7. 민감도가 `secret` 이상이면 local-only로 보냅니다.
8. Copilot route가 허용되면 runtime token을 확인합니다.
9. Copilot 응답을 받아 JSON으로 반환합니다.

## 4. 로컬과 퍼블릭의 차이

### 4.1 로컬
- host: `127.0.0.1`
- 기본 포트: `3010`
- 인증 토큰: 선택 사항
- CORS: 로컬 origin만 허용해도 충분한 경우가 많음
- 용도: HTML 리포트, 로컬 대시보드, 사내 개인 워크스테이션

### 4.2 퍼블릭
- host: `0.0.0.0`
- 인증 토큰: 필수
- HTTPS: 필수
- CORS: Vercel 또는 실제 대시보드 도메인만 명시 허용
- 용도: 외부 접근 가능한 대시보드

## 5. DLP와 라우팅 경계

### 5.1 DLP 기본 정책
- `sanitizeAtOrAbove: medium`
- `blockAtOrAbove: high`

즉,
- 이메일, 전화번호 수준은 기본적으로 sanitize 후보
- bearer token, PAT, private key, password assignment는 block 또는 강한 차단 대상

### 5.2 민감도 기본 정책
- 기본 민감도: `internal`
- local-only 최소 민감도: `secret`

즉,
- `public`, `internal`, `confidential`은 조건부로 Copilot route 가능
- `secret`는 local runner가 없으면 실패하게 설계하는 편이 안전

## 6. OpenClaw와의 관계
이 문서에서 OpenClaw는 기본 실행 경로가 아닙니다.
- standalone package가 먼저 시도합니다.
- standalone auth store에 인증이 없을 때만 OpenClaw auth store를 호환 fallback으로 읽습니다.
- 따라서 현재 기준 표현은 `OpenClaw 경유는 호환 경로`입니다.

## 7. 비용 해석 원칙
- 실측에서 quota delta가 0이면 `무차감 관측`으로 적습니다.
- 이것을 곧바로 `정책상 무제한 확정`으로 적지 않습니다.
- GitHub 정책과 third-party agent 분류는 바뀔 수 있으므로 문서에는 `정책상 무제한 확정 불가`라고 분리 표기합니다.
